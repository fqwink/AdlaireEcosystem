# Docker Verification Reports

このフォルダは、Docker実運用想定検証のログ、バグ、デバッグ情報をリポジトリ内でレポート方式により集約する場所である。

## Role

- Docker実運用想定検証の経過を記録する
- 72時間以上の長期間検証の開始、継続、停止、結果を記録する
- バグ発生時の事象、影響、再現条件、対応状況を記録する
- デバッグ情報、確認ログ、追加シナリオ結果を記録する
- Docker開発検証とDocker実運用想定検証を混同しない

## Report

Docker実運用想定検証のレポートは次へ集約する。

```text
Docker/verification/production-operation-report.md
```

## Rules

- 実行中の可変ログやSQLite確認用データはDocker volumeに残してよい。
- リポジトリには、検証状態、確認ログ、バグ、デバッグ情報をレポート方式で記録する。
- Docker実運用想定検証が72時間以上に達していない場合は、進行中または未完了として記録する。
- Docker開発検証の結果を記録する場合は、Docker開発検証として明記する。
- 停止指示を受けた場合は、停止時刻、停止理由、最終状態をレポートへ追記する。

## Production Operation Verification Scope

Docker実運用想定検証は、BaaS機能全てとCore基盤を検証範囲に含める。

Docker実運用想定検証は次の14カテゴリで行う。

1. 稼働継続検証
2. HTTP経由動作検証
3. SQLite永続化検証
4. Realtime Database検証
5. Authentication / Authorization検証
6. BaaS Admin Dashboard検証
7. Event Log内部Core基盤検証
8. Core Boundary検証
9. Operational Evidence検証
10. ドキュメント整合性検証
11. 禁止構成検証
12. 外部依存禁止検証
13. 追加シナリオ検証
14. レポート検証

運用ルール:

- Docker実運用想定検証は72時間以上を基準とする。
- 検証継続中は5分間隔の内部チェックを行う。
- 検証継続中は20分ごとにレポートを更新する。
- 追加シナリオを実施した場合はレポートへ追記する。
- バグや異常が出た場合はレポートへ記録し、バグ修正ゼロ方針に従う。
- Docker開発検証とDocker実運用想定検証は混同しない。

## Periodic Update

レポートは検証継続中に20分ごとに更新する。

更新対象:

- 継続状態
- 最新ログ
- 経過時間
- バグ記録
- デバッグ記録
- 追加シナリオ結果
- 停止時の最終結果

更新タイミング:

- ユーザーから状況確認、ログ確認、レポート更新の指示があった時
- バグ、デバッグ、追加検証を行った時
- 停止指示を受けた時
- 重要な状態変化があった時
